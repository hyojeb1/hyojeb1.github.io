## 1. `.uasset` 변경 사항 확인하기 (꿀팁)
깃허브 웹에서는 보이지 않지만, **언리얼 에디터 내부의 'Diff' 도구**를 사용하면 전후 차이를 시각적으로 볼 수 있습니다.

* **방법:** 에디터 하단의 `Content Browser`에서 해당 에셋 우클릭 -> `Source Control` -> `Diff Against Depot` (혹은 이전 리전과 비교)
* 이렇게 하면 블루프린트 노드가 어떻게 바뀌었는지, 혹은 디테일 패널의 수치가 어떻게 변했는지 전용 창으로 띄워줍니다. `BP_TPSPlayer`나 `ABP_TPS`를 볼 때 필수입니다.

## 2. 커밋 로그 기반 '3단계 복습 계획'

제공해주신 커밋들을 연관성 있는 **3개의 덩어리**로 나누어 복습하는 것이 머릿속에 지도를 그리기에 좋습니다.

### [Chunk 1] 무기 시스템 리팩터링 및 FX
> **관련 커밋:** decal 생성, 총알 이펙트/데미지/히트 처리, 총으로 발사 이벤트 변경

* **핵심 로직:** 기존에 `Player` 클래스에 몰려있던 비대한 코드를 `Gun` 클래스로 떼어내는 **관련성 분리(Decoupling)** 과정입니다.
* **복습 포인트:**
    1.  `Player`에서 `Gun`의 함수를 호출할 때 어떤 인터페이스나 포인터를 썼는지 확인하세요.
    2.  `LineTrace` 결과값(`FHitResult`)을 이용해 `SpawnEmitterAtLocation`(파티클)과 `SpawnDecalAtLocation`(데칼)을 호출하는 코드의 흐름을 따라가 보세요.
    3.  데칼의 경우, **LifeSpan**이나 **Fade** 설정을 어떻게 했는지 에디터에서 확인하세요.

### [Chunk 2] 애니메이션과 노티파이 (거미 예제)
> **관련 커밋:** 애니메이션 몽타주, 애님노티파이, 노티파이 스테이트, ABP_TPS 업데이트

* **핵심 로직:** C++에서 애니메이션의 특정 시점을 감지하거나(Notify), 일정 구간 동안 상태를 유지(Notify State)하는 법입니다.
* **복습 포인트:**
    1.  **C++ vs Blueprint:** 노티파이를 C++ 클래스로 만들었는지, 아니면 블루프린트에서 바로 호출했는지 확인하세요.
    2.  **몽타주(Montage):** 공격 애니메이션 등을 재생할 때 `PlaySlotAnimationAsDynamicMontage` 혹은 일반 몽타주 재생 함수 중 무엇을 썼는지 확인하세요.
    3.  **애니메이션 블렌딩:** C++에서 `NativeUpdateAnimation`을 통해 변수값을 넘겨주고, 이것이 애님 그래프(AnimGraph)에서 어떻게 포즈를 섞는지 흐름을 그리세요.



### [Chunk 3] AI 시스템 및 길 찾기
> **관련 커밋:** AI, BT 작업, AI Controller, 네브 매쉬

* **핵심 로직:** NPC가 스스로 판단하고 움직이게 하는 구조입니다.
* **복습 포인트:**

    3.  **NavMesh:** 에디터에서 `P` 키를 눌러 초록색 영역을 확인하고, 장애물 배치 시 실시간으로 구워지는지(Dynamic) 설정을 살펴보세요.

## 3. 추천 복습 루틴: "백지에서 다시 짜보기"

가장 좋은 복습은 코드를 눈으로 읽는 게 아니라, **아무것도 없는 빈 프로젝트나 이전 커밋 상태에서 똑같이 구현해 보는 것**입니다.

1.  **커밋 체크아웃:** `git checkout 4140d4b` (가장 초기 커밋)으로 돌아가서 오늘 배운 기능을 하나씩 직접 다시 타이핑해 보세요.
2.  **주석 달기:** 특히 C++ 코드에서 언리얼 매크로(`UFUNCTION`, `UPROPERTY`)가 왜 쓰였는지, `EditAnywhere`와 `BlueprintReadOnly`의 차이가 무엇이었는지 자신만의 언어로 주석을 달아보세요.
3.  **에러 마주하기:** 다시 짜다 보면 분명 에디터 세팅을 까먹어서 에러가 날 겁니다. 그때 깃허브의 완성된 코드를 "컨닝"하는 과정에서 진짜 실력이 늘어납니다.

# 당신에게 맞는 복습 방법

지금 상태에서는 **“무엇을 했는지 헷갈린다”**가 핵심 문제라서, 목표를 **재현 중심**으로 두는 게 맞습니다.

## 복습 단위는 “설명”이 아니라 “재구현”

각 묶음마다 아래 3가지만 적으면 됩니다.

### 템플릿

1. **목적**

   * 이 커밋 묶음으로 뭘 가능하게 했는가?

2. **핵심 변경점**

   * 어느 클래스/블루프린트가 책임을 갖게 되었는가?
   * 이벤트는 어디서 발생하고 어디로 전달되는가?

3. **내 손으로 재현**

   * 새 프로젝트/브랜치에서 기억만으로 다시 구현해보기
   * 막히면 그때 커밋 보기

이 방식이 좋은 이유는, 지금 필요한 게 “정보 회수”가 아니라 **구조 기억 복원**이기 때문입니다.

아래 질문에 답할 수 있으면 성공입니다.

* 총알 발사는 누가 시작하는가?
* 라인트레이스는 어디서 하는가?
* 히트했을 때 데칼은 어디서 생성하는가?
* 데미지 타입은 왜 따로 만들었는가?
* 플레이어에 있던 발사 로직을 총 클래스로 옮긴 이유는 무엇인가?

이때 커밋은 `4140d4b → 04f15d4 → 773b0b5`만 봅니다.
즉, **“보이는 효과” → “피해 처리” → “책임 분리”** 순서로 이해하면 됩니다. `773b0b5`에서 `TPSPlayer.cpp`에서 대량 삭제가 난 건 복습 포인트로 매우 좋습니다. “왜 삭제했는가?”를 설명해보면 설계가 정리됩니다. ([GitHub][6])

# `uasset`은 git diff로 어떻게 볼까?

정확히 말하면, **GitHub의 텍스트 diff만으로는 블루프린트 `uasset` 내용을 읽기 어렵습니다.** 실제로 해당 커밋들에서도 `ABP_TPS.uasset`, `BP_TPSPlayer.uasset`, `AM_Player.uasset` 등은 “Binary file not shown”으로 표시됩니다. ([GitHub][3])

그래서 방법은 두 가지입니다.

## 방법 1. 언리얼 에디터의 Diff 툴 사용

언리얼은 에셋/블루프린트 비교용 **UE Diff Tool**을 제공하고, 콘텐츠 브라우저에서 두 에셋을 선택해 **Diff Selected**로 비교할 수 있습니다. 소스 컨트롤과 연동된 경우 이전 버전과 비교도 지원합니다. Epic 문서도 이 방식을 안내합니다. ([Epic Games Developers][8])

이게 가장 좋습니다.
왜냐하면 당신이 궁금한 건 “바이너리가 바뀌었다”가 아니라,

* 어느 노드가 추가됐는지
* 어느 변수 default가 바뀌었는지
* 어느 이벤트 그래프 연결이 달라졌는지

이기 때문입니다.

## 방법 2. 커밋 기준으로 checkout해서 에디터에서 직접 비교

예를 들면:

* `3ec5147` 체크아웃해서 ABP/BP 상태 확인
* `cd12813` 체크아웃해서 다시 확인
* 둘을 에디터에서 비교

이렇게 하면 GitHub에서 안 보이는 블루프린트 차이를 **시각적으로 복원**할 수 있습니다. Epic 쪽 자료도 Git과 블루프린트 Diff Tool을 함께 쓰는 흐름을 설명합니다. ([Epic Games Developers][9])

## 현재 보이는 현상

스크린샷 기준으로는:

1. **플레이어가 의도한 스폰 지점이 아니라 월드 원점(0,0,0) 부근에 붙음**
2. **카메라가 플레이어 추종 카메라가 아니라, 월드 정면을 보는 고정 뷰처럼 보임**
3. 그래서 체감상 **“캐릭터 + 컨트롤러 + 카메라 체인”이 끊긴 상태**에 가깝습니다. 

즉, 이건 대개 `Move/Aim` 코드 한 줄이 망가진 문제가 아니라, 아래 셋 중 하나입니다.

* **GameMode / Default Pawn / PlayerController / Auto Possess 설정 문제**
* **맵의 PlayerStart / 배치된 Pawn / Spawn 충돌 처리 문제**
* **BP가 C++ 기본값을 덮어쓴 문제**

그리고 네 코드만 보면, 말씀대로 **“5분 전까지 멀쩡했다면 코드 원인일 확률은 낮음”** 입니다.
지금 증상은 **에디터/블루프린트/월드 세팅이 바뀌었을 때** 훨씬 자주 나옵니다. 

---

## 먼저 결론: 가장 우선 의심할 것

우선순위대로 보면 이겁니다.

### 1) BP_SU_Player가 C++ 생성자 값을 덮어썼는지

지금 C++에서는 분명히 이렇게 되어 있습니다.

* `bUseControllerRotationYaw = false`
* `CharacterMovement->bOrientRotationToMovement = true`
* `CamBoom->bUsePawnControlRotation = true`
* `FollowCam->bUsePawnControlRotation = false`

이 값들이 **BP_SU_Player에서 예전 값으로 덮어써져 있으면**, 코드가 멀쩡해도 실제 실행 결과는 완전히 달라집니다.

특히 예전에 네가 주석에 적은 것처럼
`bUseControllerRotationYaw = true`, `bOrientRotationToMovement = false` 조합을 썼던 적이 있으니,
**블루프린트가 그 예전 세팅을 캐시하고 있을 가능성**이 큽니다.

### 2) 지금 플레이 중인 Pawn이 진짜 ASU_Player / BP_SU_Player가 맞는지

이게 매우 중요합니다.

증상상:

* 카메라가 정면 고정
* 원점 근처
* 내가 만든 캐릭터 느낌이 안 남

이면 실제로는

* 다른 Pawn이 스폰됐거나
* SpectatorPawn이 잡혔거나
* 맵에 배치된 다른 액터를 보고 있거나
* CameraActor가 ViewTarget이 되었을 가능성

이 있습니다.

### 3) PlayerStart가 깨졌거나, 스폰 실패 후 fallback이 났는지

플레이어가 0,0,0에 박히는 건 흔히 다음 케이스입니다.

* `PlayerStart`가 삭제/이동/회전/충돌 문제
* 스폰 지점이 막혀서 정상 스폰 실패
* GameMode에서 DefaultPawnClass가 잘못돼서 다른 Pawn 생성
* 맵에 이미 배치된 Pawn을 possess하지 못함

### 4) GameMode / World Settings가 바뀌었는지

“방금 전까진 됐는데 갑자기 안 된다”에서 진짜 자주 나오는 원인입니다.

검토할 것:

* `World Settings -> GameMode Override`
* 프로젝트 기본 GameMode
* 해당 GameMode의 `Default Pawn Class`
* `Player Controller Class`
* PIE(Play In Editor) 설정

## 왜 코드 문제 같지 않냐

네 `ASU_Player` 코드만 놓고 보면, 카메라가 “정면 고정”으로 굳을 직접 원인은 약합니다.

핵심만 보면:

```cpp
CamBoom->bUsePawnControlRotation = true;
FollowCam->bUsePawnControlRotation = false;
```

이 조합이면 **컨트롤러 회전이 SpringArm으로 전달**되어야 합니다.

그리고 `Aim()`도 정상입니다.

```cpp
AddControllerPitchInput(Direction.Y);
AddControllerYawInput(Direction.X);
```

즉, 마우스/패드 입력이 먹고 있고, **이 Pawn이 실제로 Possess된 상태라면 카메라는 돌아야 정상**입니다.

따라서 지금 증상은 보통:

* 이 Pawn이 possess 안 됨
* 아예 다른 액터를 보고 있음
* BP에서 카메라/붐/회전 옵션이 덮어씀
* Input Mapping / Enhanced Input Context가 빠져서 `Aim()`이 안 불림

중 하나입니다.

## A. “내가 조종 중인 액터가 ASU_Player가 맞는가?”

이게 1번입니다.

`BeginPlay()`에 바로 찍으세요.

```cpp
void ASU_Player::BeginPlay()
{
	Super::BeginPlay();

	UE_LOG(LogTemp, Warning, TEXT("ASU_Player BeginPlay: %s"), *GetName());

	if (AController* C = GetController())
	{
		UE_LOG(LogTemp, Warning, TEXT("Controller: %s"), *C->GetName());
	}
	else
	{
		UE_LOG(LogTemp, Error, TEXT("Controller is nullptr"));
	}

	UE_LOG(LogTemp, Warning, TEXT("ActorLocation: %s"), *GetActorLocation().ToString());

	EquipItem(DefalutWeapon);
}
```

### 여기서 확인할 것

* 로그가 안 찍히면: **애초에 ASU_Player가 스폰되지 않음**
* Controller가 null이면: **Possess 안 됨**
* Location이 `(0,0,0)` 근처면: **스폰 위치 문제**

## B. “카메라가 내 Pawn 카메라인가?”

`BeginPlay()`에서 이것도 찍으세요.

```cpp
if (FollowCam)
{
	UE_LOG(LogTemp, Warning, TEXT("FollowCam exists"));
}
if (CamBoom)
{
	UE_LOG(LogTemp, Warning, TEXT("CamBoom exists / UsePCR=%d"), CamBoom->bUsePawnControlRotation);
}
UE_LOG(LogTemp, Warning, TEXT("bUseControllerRotationYaw=%d"), bUseControllerRotationYaw);
UE_LOG(LogTemp, Warning, TEXT("bOrientRotationToMovement=%d"), GetCharacterMovement()->bOrientRotationToMovement);
```

여기 값이 C++ 예상값과 다르면 **BP override**입니다.

## C. “Aim 입력이 실제로 들어오나?”

`Aim()`에 로그 한 줄만 넣으면 바로 판별됩니다.

```cpp
void ASU_Player::Aim(const FInputActionValue& Value)
{
	FVector2D Direction = Value.Get<FVector2D>();
	UE_LOG(LogTemp, Warning, TEXT("Aim X=%f Y=%f"), Direction.X, Direction.Y);

	AddControllerPitchInput(Direction.Y);
	AddControllerYawInput(Direction.X);
}
```

### 판별

* 로그가 안 뜬다 → **Enhanced Input Context / IA 바인딩 / 입력 asset 문제**
* 로그는 뜬다 → 입력은 살아있음, 그럼 **ViewTarget / Possession / Camera 세팅 문제**

* 현재 맵의 `GameMode Override`가 무엇인지

* 레벨 시퀀스나 카메라 액터가 뷰를 잡고 있지 않은지

* 존재하는지
* 바닥/벽에 박혀 있지 않은지

* 여러 개 있을 때 다른 PlayerStart가 선택되는 상황 아닌지

특히 이 항목들:

* `Use Controller Rotation Yaw`
* `CharacterMovement > Orient Rotation To Movement`
* `SpringArm > Use Pawn Control Rotation`
* `Camera > Use Pawn Control Rotation`
* `SpringArm Length`
* `Inherit Pitch / Yaw / Roll`
* `Pawn > Auto Possess Player`

여기서 **노란 화살표(Reset to Default)**가 보이면, 블루프린트가 C++ 기본값을 덮어쓰고 있다는 뜻입니다.

## 우선 확인 순서

### 1. `Required` 모듈의 `Material`

스크린샷 기준으로 각 emitter의 **Required** 모듈이 보입니다.
여기서 파티클이 직접 참조하는 건 보통 **Texture가 아니라 Material**입니다.

먼저 확인할 것:

* 각 emitter의 **Required → Material**
* 그 머티리얼이 `None`으로 바뀌었는지
* 다른 emitter는 정상인데 특정 emitter만 비어 있는지

파티클 텍스처 소실은 대개 여기서 시작합니다.

---

### 2. 머티리얼 안에서 참조하는 `Texture Sample`

`Required`에 머티리얼이 살아 있어도, 그 머티리얼 내부의 Texture 경로가 깨졌을 수 있습니다.

확인:

* 머티리얼 열기
* `Texture Sample`, `Texture Object`, `Texture Parameter`가 `None`인지
* 머티리얼 인스턴스면 **부모 머티리얼**과 **파라미터 오버라이드**도 같이 보기
* SubUV라면 **ParticleSubUV** 노드 / SubUV용 텍스처가 비어 있지 않은지

즉 구조는 보통 이렇습니다.

`Particle System -> Required.Material -> Material/MI -> Texture`

---

### 3. `Reference Viewer`

경로 깨짐이면 가장 빨리 잡히는 게 이겁니다.

확인 대상:

* 파티클 에셋 우클릭 → **Reference Viewer**
* 머티리얼 우클릭 → **Reference Viewer**
* 문제 텍스처 우클릭 → **Reference Viewer**

여기서 보면:

* 참조가 끊겼는지
* Redirector만 남았는지
* 다른 폴더로 이동된 뒤 재연결이 안 됐는지
  한 번에 파악됩니다.

---

### 4. `Fix Up Redirectors`

폴더 이동/이름 변경 후 가장 흔한 원인입니다.

확인:

* Content Browser에서 상위 폴더 우클릭
* **Fix Up Redirectors in Folder**

특히 이런 경우 거의 의심해볼 만합니다.

* 텍스처/머티리얼 폴더를 옮겼다
* 마이그레이션 후 경로가 바뀌었다
* Git/SVN 머지 이후 일부 참조만 꼬였다

### 5. 로그에서 로드 실패 메시지

에디터가 이미 답을 주는 경우가 많습니다.

봐야 할 로그:

## 파티클에서 특히 자주 놓치는 지점

### SubUV 사용하는 경우

스크린샷상 `SubImage Index` 모듈이 보입니다.
이 경우 단순 Texture가 아니라 **SubUV용 머티리얼/텍스처 설정**이 맞아야 합니다.

확인:

* 머티리얼이 `ParticleSubUV` 기반인지
* 텍스처가 sprite sheet인지
* Required 쪽 SubUV 관련 설정이 살아 있는지

텍스처만 살아 있어도 머티리얼 세팅이 틀리면 결과적으로 “사라진 것처럼” 보입니다.

---

### 머티리얼 Usage 플래그

경로 문제와 별개지만 같이 확인할 가치가 있습니다.

머티리얼에서:

* **Used with Particle Sprites**
* 필요한 경우 **Used with Beam Trails / Mesh Particles**

이게 꺼져 있으면 렌더가 이상하게 보일 수 있습니다.

---

### 머티리얼 인스턴스 체인

실무에서 자주 놓칩니다.

예:

* Particle이 `MI_MuzzleFlash` 참조
* `MI_MuzzleFlash`의 부모 `M_MuzzleFlash`는 살아 있음
* 그런데 `MI`의 Texture Parameter만 비어 있음

겉으로 보면 “파티클 경로 깨짐”처럼 보이는데 실제론 **인스턴스 파라미터 유실**입니다.

# 가장 안전한 연결 방법

## 1. `P_MuzzleFlash` 열기

* Content Browser에서 `P_MuzzleFlash` 더블클릭
* Cascade 열기

---

## 2. emitter별로 `Required` 모듈 확인

각 emitter 컬럼에서 맨 위쪽의 **Required** 클릭

왼쪽 `Details` 패널에서 확인:

* **Material**
* 필요한 경우 **Screen Alignment**, **SubUV 관련 옵션**

여기서 `None` 이거나 깨진 상태일 겁니다.

### 해야 할 일

`Material` 슬롯에 현재 살아있는 새 경로의 머티리얼을 다시 넣기

예상 매핑은 대략 이렇습니다.

* flash/sprite 계열 → `M_muzzle_flash_subUV`
* glow 계열 → `M_MuzzleGlow`
* lens flare 계열 → `M_Lensflare_01`
* smoke 계열 → `M_ImpactSmoke` 또는 `M_SmokeBall`

이름으로 거의 구분 가능합니다.

## 3. mesh emitter면 Mesh도 다시 연결

로그에 이것도 빠져 있습니다.

* `SM_MuzzleCone`

즉 mesh emitter가 하나 있으면,

* emitter의 **TypeData** 또는 mesh 관련 모듈 클릭
* `Mesh` 슬롯에 `SM_MuzzleCone` 다시 지정
* 필요하면 그 emitter의 `Required.Material`도 함께 재지정

보통 muzzle flash 파티클에는 cone mesh emitter가 하나 섞여 있을 수 있습니다.

---

## 4. SubUV emitter 확인

스크린샷상 `SubImage Index`가 있는 emitter가 보였으니, 이 emitter들은 단순 sprite가 아니라 **SubUV 머티리얼**을 써야 합니다.

확인:

* `Required.Material` → `M_muzzle_flash_subUV` 같은 SubUV용 머티리얼 지정
* `SubImage Index` 모듈 유지
* `T_explo_SubUV` 같은 sprite sheet 텍스처가 머티리얼 안에 정상 연결되어 있는지 확인

즉, 단순히 텍스처만 있는 걸로는 안 되고 **머티리얼이 그 텍스처를 물고 있어야** 합니다.

# 더 좋은 방법: 폴더를 원래 경로로 되돌리기

지금처럼 `HuiunArchive` 밑에 넣으면 기존 참조가 전부 깨집니다.
그래서 가능하면 **원래 기대하던 경로 그대로 복원**하는 게 제일 깔끔합니다.

즉 실제 파일을 이런 구조로 두는 게 좋습니다.

* `Content/Effects/Materials/LensFlare/M_Lensflare_01`
* `Content/Effects/Materials/Ramp/M_SoftGradient_Muzzle`
* `Content/Effects/Materials/Smoke/M_ImpactSmoke`
* `Content/Effects/Materials/Smoke/M_SmokeBall`
* `Content/Effects/Materials/Weapon/M_muzzle_flash_subUV`
* `Content/Effects/Materials/Weapon/M_MuzzleGlow`
* `Content/Effects/Meshes/Weapon/SM_MuzzleCone`

이렇게 **원래 위치로 넣으면** 기존 참조가 자동으로 살아날 가능성이 큽니다.

## 추천 이유

파티클 하나만의 문제가 아닐 수 있습니다.
다른 이펙트, 블루프린트, 머티리얼 인스턴스도 `/Game/Effects/...` 를 참조할 수 있어서,
수동 재연결보다 **원래 폴더 구조 복구**가 훨씬 안정적입니다.

---

# 그래서 실무적으로는 둘 중 하나입니다

## 선택 1: 원래 폴더 구조로 복원

가장 권장.

* `HuiunArchive`에 넣은 에셋들을
* 원래 로그에 나온 `/Game/Effects/...` 구조로 이동
* 그 다음 **Fix Up Redirectors in Folder**
* 에디터 재시작 또는 에셋 다시 열기

이 경우 자동 복구될 가능성이 높습니다.

---

## 선택 2: 현재 `HuiunArchive` 유지하고 수동 재연결

가능은 하지만 귀찮고 누락 위험이 큽니다.

* `P_MuzzleFlash` 안의 각 emitter
* 관련 머티리얼 인스턴스
* mesh emitter
  전부 수동으로 다시 연결해야 합니다.

## 핵심 정리

`.uasset`가 바이너리라는 점도 맞지만, 더 정확히는 **언리얼 에셋 참조가 “파일명”이 아니라 “오브젝트 경로”를 기준으로 저장된다**는 게 핵심입니다.

예를 들면 파티클 안에는 대충 이런 식의 참조가 들어 있습니다.

* `/Game/Effects/Materials/Weapon/M_MuzzleGlow`
* `/Game/Effects/Materials/Weapon/M_muzzle_flash_subUV`

여기서 에셋을 `HuiunArchive/Effects/...` 쪽으로 넣어버리면 실제 경로가 이렇게 바뀝니다.

* `/Game/HuiunArchive/Effects/Materials/Weapon/M_MuzzleGlow`

그러면 언리얼은 이걸 **같은 파일**로 보지 않습니다.
이름이 같아도 **경로가 다르면 다른 에셋**입니다.

---

## 왜 바로 깨졌는가

`P_MuzzleFlash`는 원래 강사님 프로젝트 구조 기준으로 만들어진 에셋이라서, 내부 참조도 전부 `/Game/Effects/...` 를 보고 있었던 겁니다.

그래서 `Effects` 폴더를 새로 만들어서 `Content` 바로 아래에 맞춰 넣으니까:

* 참조 경로와
* 실제 에셋 위치가

다시 일치해서 정상 복구된 겁니다.

---

## 실무적으로 기억할 것

### 1. 외부에서 받은 언리얼 에셋은 폴더 구조째 유지

가장 중요합니다.

* 상대가 준 `Content/Effects/...`
* 상대가 준 `Content/Blueprints/...`

이 구조를 **그대로** 넣는 게 안전합니다.

임의로

* `MyArchive`
* `HuiunArchive`
* `Temp`
  같은 최상위 폴더를 하나 더 감싸면 참조가 깨질 수 있습니다.

---

### 2. 탐색기 복붙보다 Content Browser 이동이 안전

이미 프로젝트 안에 들어온 에셋을 옮길 때는

* 가능하면 **언리얼 Content Browser에서 Move**
* 이후 **Fix Up Redirectors**

이 루트가 제일 안전합니다.

탐색기에서 직접 폴더 이동하면 레퍼런스/리다이렉터 관리가 꼬일 수 있습니다.

---

### 3. “이름이 같은데 왜 못 찾지?”는 거의 경로 문제

언리얼에서는 흔합니다.

특히 로그에 이런 게 뜨면 거의 확정입니다.

* `Failed to load`
* `dependent package ... was not available`
* `does not exist on disk`
* `Skipped package ... has a valid mounted mount point but does not exist`

이건 대부분:

* 경로 변경
* sync 누락
* 폴더 구조 불일치
  중 하나입니다.

# 먼저 전체 그림

언리얼에서 이 네 가지는 역할이 다릅니다.

## 역할 구분

* **Player(Character/Pawn)**
  입력을 받고, 게임플레이 의도를 결정합니다.
  예: 발사한다, 리로드한다, 공격한다.

* **AnimInstance / Animation Blueprint**
  현재 캐릭터가 **어떤 자세와 애니메이션 상태**에 있어야 하는지 계산합니다.
  속도, 점프 여부, 조준 여부, 죽음 여부 같은 값을 바탕으로 포즈를 만듭니다. 애니메이션 블루프린트는 복잡한 애니메이션 행동을 제어하는 용도입니다. ([Epic Games Developers][1])

* **Montage**
  “지금 당장 이 액션 애니메이션을 재생하라”에 가깝습니다.
  상태머신 위에 덮어서 재생하는 **명령형 액션 재생 도구**라고 보면 됩니다. Epic 문서도 몽타주를 블루프린트로 제어 가능한 애니메이션 에셋으로 설명합니다. ([Epic Games Developers][2])

* **Notify / Notify State**
  애니메이션 타임라인 특정 시점 또는 구간에 맞춰 이벤트를 발생시키는 장치입니다. Notify는 한 시점, Notify State는 시작-종료를 가지는 구간 이벤트입니다. ([Epic Games Developers][3])

# 당신이 이해해야 하는 핵심 한 문장

## 코드 지연이 아니라 애니메이션 프레임에 맞춘다

당신이 이미 적은 설명:

> 행동 타이밍을 코드 지연으로 맞추는 대신 애니메이션 프레임 이벤트에 연결했다.

이 설명은 정확합니다.
조금 더 실무적으로 바꾸면 이렇습니다.

> **게임플레이 의사결정은 Player가 하고, 실제 발사/타격/이펙트 시점은 애니메이션 타임라인의 Notify가 확정한다.**

즉:

* Player: “발사 시작”
* Montage: “발사 동작 재생”
* Notify: “정확히 이 프레임에서 총알 발사”
* Notify State: “이 구간 동안 공격 판정 활성화”

이 구조입니다. ([Epic Games Developers][2])

# 몽타주를 왜 쓰는가

## 몽타주는 “액션성 동작”에 적합합니다

몽타주는 여러 애니메이션을 하나의 에셋 안에서 조합하고, 런타임에 블루프린트나 코드로 재생을 제어할 수 있게 해 줍니다. 섹션, 슬롯, 블렌딩 등으로 제어 가능합니다. ([Epic Games Developers][2])

실무 감각으로는 이렇게 보면 됩니다.

### 상태머신이 잘하는 것

* Idle / Walk / Run
* Falling / Landing
* 계속 반복되는 이동 상태

### 몽타주가 잘하는 것

* 발사
* 리로드
* 근접 공격
* 피격 리액션
* 스킬 시전

왜냐하면 이런 동작은 **“지금 한 번 실행하라”**는 성격이 강하기 때문입니다.
이동처럼 계속 유지되는 상태가 아니라, **명령에 의해 시작되고 끝나는 액션**입니다. ([Epic Games Developers][2])

# Notify와 Notify State의 차이

## Notify

한 점 이벤트입니다.

예:

* 발사 순간
* 발소리 발생 순간
* 탄피 배출 순간
* 파티클 재생 순간

즉, “이 프레임에 한 번 실행”입니다. Epic 문서도 노티파이를 애니메이션 시퀀스에 동기화된 이벤트를 보내고 받는 시스템으로 설명합니다. ([Epic Games Developers][3])

## Notify State

구간 이벤트입니다.

예:

* 검 휘두르는 동안만 히트박스 활성화
* 무적 시간 구간
* 차징 구간
* 특정 IK 보정 활성 구간

즉:

* Begin
* Tick
* End

세 단계가 있습니다. `UAnimNotifyState` API도 이 구조를 전제로 합니다. ([Epic Games Developers][4])

## 한 줄 요약

* **Notify = 점**
* **Notify State = 구간**

# 당신이 막힌 핵심: AnimInstance와 Player 사이 데이터 전달 경로

여기가 가장 중요합니다.

## 정답은 “양방향 가능하지만, 주도권은 분리해야 한다”입니다

### 1. Player → AnimInstance

이 방향은 **상태 전달**입니다.

예:

* `bIsAiming`
* `bIsDead`
* `Speed`
* `bIsInAir`
* `bIsReloading`

보통 AnimInstance는 매 프레임 캐릭터를 참조해서 필요한 값을 읽습니다.
즉, AnimInstance가 `Try Get Pawn Owner` 또는 캐릭터 캐스팅을 통해 Player 상태를 가져와 애니메이션 블루프린트 변수로 반영합니다. 애니메이션 블루프린트는 이런 식으로 포즈 계산에 필요한 데이터를 받아 사용합니다. ([Epic Games Developers][1])

실무적으로는 이렇게 이해하면 됩니다:

> **애니메이션 블루프린트는 캐릭터의 상태를 읽어서 포즈를 만든다.**

이건 “상태 기반”입니다.

---

### 2. Player → AnimInstance → Montage

이 방향은 **재생 명령**입니다.

예:

* 발사 입력 받음
* Player가 `PlayAnimMontage(FireMontage)` 또는 AnimInstance를 통해 몽타주 재생
* 몽타주가 재생되기 시작함

즉:

```text
입력
→ Player의 Fire() 호출
→ FireMontage 재생
→ 애니메이션 진행
→ Notify 시점 도달
```

몽타주 재생은 캐릭터/애님 인스턴스에서 제어할 수 있고, 런타임 제어가 핵심입니다. ([Epic Games Developers][2])

---

### 3. AnimInstance / Notify → Player

이 방향은 **타이밍 콜백**입니다.

예:

* 총 쏘는 손동작의 정확한 프레임에서 Notify 발생
* Notify가 Player 또는 Weapon에게 “지금 발사하라” 전달
* 실제 Trace / Spawn Emitter / Ammo 감소 수행

이게 당신이 배우려는 핵심입니다.

> **행동 시작은 Player가 하지만, 행동 확정 타이밍은 Notify가 준다.**

즉, Player가 “쏜다”를 결정하고, Notify가 “지금 쏴라”를 알려줍니다. ([Epic Games Developers][3])

---

# 실전 구조로 그리면 이렇게 됩니다

## 발사 예시

```text
마우스 클릭
→ Player::InputFire()
→ FireMontage 재생
→ 몽타주 안의 FireNotify 시점 도달
→ Notify가 Player/Weapon 함수 호출
→ 실제 발사 처리(Trace, Damage, Decal, Particle)
```

이 구조의 장점:

* 코드에서 `Delay(0.23초)` 같은 임시 처리가 사라짐
* 애니메이션이 빨라지거나 바뀌어도 타이밍이 계속 맞음
* 리로드/근접공격/스킬에 재사용 가능

# 왜 AnimInstance와 Player를 헷갈리기 쉬운가

그 이유는 둘 다 애니메이션에 관여하기 때문입니다.
하지만 역할은 다릅니다.

## Player가 해야 할 일

* 입력 받기
* 무기/탄약/데미지 같은 게임플레이 로직 결정
* 몽타주 재생 지시
* Notify가 왔을 때 실제 게임 로직 수행

## AnimInstance가 해야 할 일

* 현재 캐릭터 상태를 읽어 포즈 계산
* 상태머신 전환
* 블렌드
* 필요한 경우 몽타주 재생 보조
* Notify 이벤트를 받을 수 있는 애니메이션 레이어 역할

즉:

> **Player는 게임플레이의 주체, AnimInstance는 포즈 계산의 주체**

이렇게 나누면 됩니다. ([Epic Games Developers][1])

# 거미 예제를 어떻게 봐야 하나

거미 예제는 대개 “노티파이 그 자체”보다, **애니메이션과 실제 행동 타이밍이 붙는 감각**을 보여주는 용도로 보면 됩니다.

당신이 관찰해야 하는 것은 이것입니다.

## 관찰 포인트

* 공격 애니메이션이 **언제 시작되는가**
* 실제 공격 판정은 **언제 켜지는가**
* 공격 판정은 **얼마 동안 유지되는가**
* 공격이 끝난 후 상태가 **언제 복귀되는가**

이걸 기준으로 보면:

* **한 순간 공격 확정**이면 Notify
* **공격 유효 시간 구간**이면 Notify State

거미 예제는 대개 “애니메이션 재생”보다 “공격 타이밍 동기화”를 보여주는 참고 사례로 읽는 게 맞습니다.

# 지금 단계에서의 오해 3개

## 오해 1. 몽타주를 이해해야만 Notify를 쓸 수 있다

반은 맞고 반은 아닙니다.

* Notify는 시퀀스에도 넣을 수 있습니다. ([Epic Games Developers][3])
* 다만 실전에서는 액션성 동작이 몽타주로 재생되는 경우가 많아서 함께 배우게 됩니다. ([Epic Games Developers][2])

## 오해 2. AnimInstance가 게임 로직도 전부 가져야 한다

아닙니다.

AnimInstance는 게임 로직의 본체가 아니라 **애니메이션 표현 계층**에 가깝습니다.
실제 데미지, 탄약 감소, 무기 발사 등은 Player/Weapon 쪽에 두는 편이 더 깔끔합니다.

## 오해 3. Notify가 발사를 “대신 결정”한다

아닙니다.

발사 결정은 Player가 합니다.
Notify는 **결정된 행동의 실행 시점**만 맞춰 줍니다.

# 당신의 학습 상태에 맞는 이해 순서

지금은 “루틴”보다 아래 순서가 낫습니다.

## 1단계: 이 문장만 완전히 이해

> Player가 발사를 결정하고, Montage가 동작을 재생하며, Notify가 실제 실행 시점을 알려준다.

이 문장을 스스로 설명할 수 있어야 합니다.

## 2단계: Notify와 Notify State를 구분

* 총 쏘는 순간: Notify
* 칼 공격 판정 유지 구간: Notify State

## 3단계: AnimInstance의 역할 축소 이해

AnimInstance는 “포즈 계산기”라고 생각하십시오.
모든 걸 거기 넣으려 하면 더 헷갈립니다.

## 4단계: 거미 예제를 타이밍 관점으로만 보기

“거미가 어떻게 공격했나”보다
“공격이 시작되고, 유효 구간이 있고, 끝나는 타이밍이 어디에 묶였나”를 봐야 합니다.

```cpp
	AddMovementInput(NewForwardVector * Direction.Y);
	AddMovementInput(NewRightVector * Direction.X);
```

# 의미

```text
LogParticles: Warning: Trail Notify: Null PSTemplate for trail notify in anim: /Game/Characters/Mannequins/Montage/AM_Reload.AM_Reload
```

# 왜 생기나

스크린샷을 보면 타임라인 아래에 `Trail` 이라는 Notify가 있습니다.
이 Notify는 보통 다음 같은 정보를 필요로 합니다.

* 어떤 파티클을 쓸지 (`PSTemplate`)
* 어느 본/소켓 사이에 트레일을 붙일지
* 시작/끝 소켓 이름

그런데 지금은 **Trail Notify만 꽂아 놓고, 실제 파티클 템플릿을 안 넣었거나**, 예전에 있던 Niagara/Cascade 자산 참조가 끊긴 상태로 보입니다.

# 지금 해야 할 것

## 1. `Trail` Notify 선택

몽타주 타임라인에서 `Trail` Notify를 클릭합니다.

## 2. Details 패널 확인

오른쪽 `Details`에서 아래 비슷한 항목을 찾습니다.

* `PSTemplate`
* `Particle System`
* `Template`

여기가 `None`이면 원인 확정입니다.

## 3. 처리 방법 선택

### 경우 A. 트레일 이펙트가 필요 없다

그 Notify를 **삭제**하면 됩니다.

### 경우 B. 트레일 이펙트가 필요하다

`PSTemplate`에 적절한 파티클 시스템을 넣어야 합니다.

단, 이건 **Cascade 기반 Trail Notify**일 가능성이 큽니다.
Niagara를 쓰는 구조라면 다른 방식으로 붙여야 할 수도 있습니다.

# 왜 리로드 몽타주에 Trail이 있나

보통 둘 중 하나입니다.

## 1. 예제 따라 하다가 테스트용 Notify가 남음

수업 중 실습 흔적일 가능성이 큽니다.

## 2. 다른 애니메이션에서 복사된 Notify가 남음

근접 무기 휘두름 같은 데서 쓰던 Trail Notify가 리로드 몽타주에 남았을 수 있습니다.

리로드에는 대개 필요한 게:

* 탄창 분리 시점
* 탄창 삽입 시점
* 장전 완료 시점

이지, **Trail Notify**는 대개 핵심이 아닙니다.

# 핵심 진단

## 지금 가장 의심되는 것

`PlayAnimMontage()`는 호출되는데, ABP 쪽 AnimGraph에 **Montage Slot**이 없어서 출력이 안 됨.

몽타주는 그냥 자동으로 재생되는 게 아니라, 보통 AnimGraph 안에 이런 흐름이 있어야 합니다.

```text
기본 Locomotion Pose
→ Slot("DefaultSlot" 또는 "UpperBody")
→ Final Animation Pose
```

또는 상체만 쓸 거면:

```text
Locomotion
→ Layered Blend Per Bone
   - Base Pose: locomotion
   - Blend Pose: Slot("UpperBody")
→ Final Pose
```

# 왜 그렇게 보냐

스크린샷의 몽타주 타임라인을 보면 슬롯 트랙 이름이 **`DefaultGroup.UpperBody`** 계열로 보입니다.
그런데 당신이 보여준 `ABP_TPS` 그래프에는 **Slot 노드가 전혀 없었습니다.**

즉:

* 몽타주는 `"UpperBody"` 슬롯으로 재생하려고 함
* ABP는 그 슬롯을 받아줄 노드가 없음
* 결과적으로 로그는 찍히고 함수는 들어가도, 실제 포즈 반영은 안 됨

이 패턴이 아주 흔합니다.

# 먼저 확인할 것 1순위

## `PlayAnimMontage` 반환값 찍기

이렇게 바꿔보세요.

```cpp
void ASU_Player::Reload()
{
    UE_LOG(LogTemp, Warning, TEXT("Reload"));

    ASU_WeaponBase* ChildWeapon = Cast<ASU_WeaponBase>(Weapon->GetChildActor());
    if (ChildWeapon)
    {
        float Result = 0.0f;

        switch (ChildWeapon->WeaponType)
        {
        case EWeaponState::Pistol:
        {
            Result = PlayAnimMontage(ReloadAnimation, 1.0f, FName("Pistol"));
            UE_LOG(LogTemp, Warning, TEXT("PlayAnimMontage Pistol Result = %f"), Result);
            break;
        }
        case EWeaponState::Rifle:
        {
            Result = PlayAnimMontage(ReloadAnimation, 1.0f, FName("Rifle"));
            UE_LOG(LogTemp, Warning, TEXT("PlayAnimMontage Rifle Result = %f"), Result);
            break;
        }
        }
    }
}
```

## 해석

* **0 이하**면 재생 실패 쪽
* **양수**면 몽타주는 재생 시도됨

양수가 뜨는데 화면에 변화가 없으면, 거의 **ABP 슬롯 문제**입니다.

# ABP에서 해야 할 것

## 방법 A. 가장 단순한 방식

AnimGraph에 `Slot` 노드를 추가합니다.

구조 예시:

```text
BlendSpacePlayer
→ RotationOffsetBlendSpace
→ Slot("UpperBody")
→ Output Pose
```

단, 이렇게 하면 전체 몸이 몽타주에 덮일 수 있습니다.

---

## 방법 B. 보통 더 많이 쓰는 방식

상체만 몽타주로 덮습니다.

```text
Locomotion Pose
→ Layered Blend Per Bone
   Base Pose = locomotion
   Blend Pose = Slot("UpperBody")
→ Final Output
```

그리고 분기 본을 예를 들면:

* `spine_01`
* `spine_02`

같이 잡습니다.

이렇게 하면:

* 하체는 이동 계속
* 상체만 리로드 몽타주 재생

이게 보통 총기 리로드에 맞습니다.

# 반드시 맞춰야 하는 것

## 슬롯 이름 일치

몽타주에 설정된 슬롯과 ABP의 Slot 노드 이름이 같아야 합니다.

스크린샷상 몽타주가 `UpperBody` 계열 슬롯을 쓰는 것 같으니,
ABP Slot 노드도 같은 슬롯명을 써야 합니다.

즉 확인할 것:

* 몽타주: `DefaultGroup.UpperBody` 또는 `UpperBody`
* ABP Slot 노드: 같은 슬롯명

이름이 다르면 재생 안 됩니다.

# 그다음으로 의심할 것들

## 1. 섹션 이름 오타

당신 코드는:

```cpp
PlayAnimMontage(ReloadAnimation, 1.0f, FName("Pistol"));
```

그래도 확인:

* 섹션 이름이 정확히 `Pistol`
* 공백 없음
* 대소문자 동일

이것도 로그로 확인 가능합니다.

```cpp
if (!ReloadAnimation)
```

## 3. Mesh의 Anim Class 문제

플레이어 스켈레탈 메시가 실제로 `ABP_TPS`를 쓰고 있는지 확인해야 합니다.

확인 위치:

* `BP_SU_Player`의 Mesh
* Animation Mode = Use Animation Blueprint
* Anim Class = `ABP_TPS`

이게 아니면 몽타주 호출해도 기대대로 안 보일 수 있습니다.

ABP가 매 프레임 강하게 포즈를 만들고 있는데, 몽타주 슬롯 경로가 없으면 그냥 locomotion이 계속 이깁니다.

# 1. Zombie는 무엇을 표현하는가?

여기서 Zombie는 **월드에 실제로 존재하는 적 캐릭터**입니다.

즉 Zombie가 맡는 것은 보통 이런 것들입니다.

* 몸체
* 위치
* 이동
* 피격/사망
* 애니메이션 재생
* AIController가 붙어서 조종할 대상

중요한 점은:

> **Zombie가 “생각”을 직접 한다기보다, 생각의 결과를 몸으로 수행하는 Pawn/Character**에 가깝다는 것입니다.

당신 자료에서도 BT는 `ControlledPawn`을 기준으로 동작하고, `GetRandomPatrolPoint`에서 순찰 원점을 `ControlledPawn`의 위치로 잡습니다. 즉 “몸체 기준 위치”는 Zombie 쪽입니다.

# 2. AIController는 무엇을 담당하는가?

AIController는 **좀비의 두뇌**라고 보면 됩니다.

이 자료 기준으로 AIController 계층은 이런 일을 한다고 이해하면 맞습니다.

* Behavior Tree 실행
* Blackboard 소유/접근
* 어떤 상태를 따라갈지 결정
* Pawn(Zombie)에게 이동/행동 지시

특히 Behavior Tree 루트가 `BB_Zombie`를 붙잡고 시작하고, 그 아래에서 `Idle`, `Chase`, `Battle`, `Death` 시퀀스를 선택하는 구조이므로, **상태 선택과 실행 흐름 관리**는 AIController + BT 쪽 책임입니다. 

한 줄로 말하면:

> **Zombie가 몸이면, AIController는 그 몸에 “지금 순찰해”, “플레이어 쫓아”, “죽음 상태로 가”를 지시하는 조종기**입니다.

# 3. Blackboard 키는 무엇을 저장하는가?

당신 자료에서 실제로 확인되는 키는 크게 3개입니다.

## `CurrentState`

이건 지금 좀비가 어떤 상태인지 저장합니다.

자료에 바로 보이는 값:

* `Idle`
* `Chase`
* `Battle`
* `Death`

그리고 Selector 아래 각 시퀀스는 `CurrentState Is Equal To ...` 데코레이터로 분기됩니다. 즉 **BT의 분기 기준 상태값**입니다. 

## `Player`

이건 추적 대상 오브젝트를 저장합니다.

`BTTask_SetPlayer`에서 `GetAllActorsOfClass(BP_TPSPlayer)`로 플레이어를 찾고, 그 첫 번째 원소를 `SetBlackboardValueAsObject`로 `PlayerKey`에 넣고 있습니다. 그리고 Chase 시퀀스의 `MoveTo`가 `BlackboardKey = Player`를 사용합니다. 즉 **쫓아갈 대상 Actor 참조**입니다. 

## `Target`

이건 순찰/이동 목표 위치를 저장합니다.

`GetRandomPatrolPoint`가 `K2_GetRandomReachablePointInRadius`로 랜덤 도달 가능 지점을 뽑고, 그 값을 `SetBlackboardValueAsVector`로 저장합니다. 그리고 Idle 시퀀스의 `MoveTo`는 `BlackboardKey = Target`을 사용합니다. 즉 **순찰 목적지 좌표(Vector)** 입니다. 

정리하면:

* `CurrentState` = 지금 AI가 어떤 모드인가
* `Player` = 누구를 쫓는가
* `Target` = 어디로 이동하는가

# 4. BTTask는 왜 필요한가?

BTTask는 **“행동 트리 안에서 실제 일을 하는 실행 단위”**입니다.

데코레이터는 “조건 검사”에 가깝고,
Task는 “실제로 값을 넣거나 움직이게 하거나 기다리는 것”입니다.

당신 자료에서 그 역할이 아주 잘 보입니다.

## `BTTask_SetPlayer`

* 플레이어를 찾음
* Blackboard의 `Player`에 저장

즉 **추적 대상을 확정하는 작업**입니다. 

## `GetRandomPatrolPoint`

* 좀비 현재 위치를 원점으로 삼음
* 반경 500 안에서 도달 가능한 랜덤 지점 탐색
* 그 위치를 Blackboard `Target`에 저장

즉 **순찰 목적지를 계산하는 작업**입니다. 

## `BTTask_SetCurrentState`

* `CurrentState`를 `Idle` 또는 `Chase` 같은 값으로 세팅

즉 **분기 기준이 되는 상태를 Blackboard에 기록하는 작업**입니다. Chase 시퀀스에는 `Set Chase`가 있고, Idle 시퀀스에도 `CurrentState`를 세팅하는 작업이 앞에 붙어 있습니다. 

따라서 BTTask가 필요한 이유는:

> **“이동한다”만으로는 부족하고, 이동 전에 ‘누굴 쫓을지’, ‘어디로 갈지’, ‘지금 상태가 뭔지’를 계산해서 Blackboard에 넣어야 하기 때문**입니다.

즉 BTTask는 BT를 **실행 가능한 로직**으로 만드는 부품입니다.

# 5. NavMesh는 이 전체 구조에서 어떤 전제 조건인가?

NavMesh는 **이 AI가 실제로 걸어 다닐 수 있게 하는 길 정보**입니다.

당신 자료에서 이게 직접 드러나는 부분은 `GetRandomPatrolPoint`입니다.

* `NavigationSystemV1`
* `K2_GetRandomReachablePointInRadius`

여기서 핵심 단어가 **Reachable**입니다.
즉 “갈 수 있는 지점”을 NavMesh 기준으로 뽑는 것입니다. 

그래서 NavMesh가 없으면:

* `GetRandomReachablePointInRadius`가 유효한 점을 못 줄 수 있음
* `MoveTo(Target)`가 실패할 수 있음
* `MoveTo(Player)`도 경로 계산이 안 될 수 있음

즉 이 구조에서 NavMesh는 선택사항이 아니라:

> **순찰과 추적이 성립하기 위한 전제 조건**

입니다.

특히 당신 BT의 Idle은

* 상태 설정
* 랜덤 순찰점 구하기
* `MoveTo(Target)`
* `Wait`

순서이고, Chase는

* 상태 설정
* `MoveTo(Player)`
* `Wait`

순서입니다. 이 두 시퀀스 모두 결국 이동 노드가 핵심이라 NavMesh가 깔려 있어야 전체가 작동합니다.

# 당신이 그대로 적어도 되는 답안

## Zombie는 무엇을 표현하는가?

월드에 배치된 실제 적 캐릭터다. 이동, 피격, 애니메이션 등 몸체 역할을 하며, AIController가 이 Zombie를 조종한다. `GetRandomPatrolPoint`도 `ControlledPawn`의 위치를 기준으로 계산한다. 

## AIController는 무엇을 담당하는가?

Behavior Tree와 Blackboard를 통해 좀비의 판단 흐름을 관리하고, Pawn에게 이동/행동을 지시하는 두뇌 역할이다. 현재 구조는 `Idle`, `Chase`, `Battle`, `Death` 중 어떤 상태를 실행할지 선택한다. 

## Blackboard 키는 무엇을 저장하는가?

`CurrentState`는 현재 AI 상태, `Player`는 추적 대상 오브젝트, `Target`은 순찰 목적지 위치를 저장한다. `MoveTo(Player)`와 `MoveTo(Target)`가 각각 이를 사용한다. 

## BTTask는 왜 필요한가?

Behavior Tree 안에서 실제 작업을 수행하기 위해 필요하다. 예를 들어 `BTTask_SetPlayer`는 플레이어를 찾아 Blackboard에 저장하고, `GetRandomPatrolPoint`는 도달 가능한 랜덤 위치를 구해 `Target`에 저장하며, `BTTask_SetCurrentState`는 현재 상태값을 기록한다. 

## NavMesh는 이 전체 구조에서 어떤 전제 조건인가?

순찰과 추적 이동이 성립하려면 NavMesh가 필요하다. `GetRandomReachablePointInRadius`와 `MoveTo`가 모두 네비게이션 가능한 영역을 전제로 동작하기 때문이다.

```text id="8um6rg"
Node0 Root
→ Node1 Selector
   → Node7 Idle
   → Node2 Chase
   → Node5 Battle
   → Node6 Death
```

입니다.

---

# 각 시퀀스 안의 의미

## 1) Idle = Node7

Node7 아래에 `Node11`, `Node9`, `Node10`이 있고, 실제 원본에선 Idle 시퀀스가 다음 4단계입니다.

```text id="p4brn0"
SetCurrentState
→ GetRandomPatrolPoint
→ MoveTo(Target)
→ Wait
```

즉:

* 현재 상태를 Idle로 기록
* 랜덤 순찰 지점(Target) 계산
* 그 위치로 이동
* 잠깐 대기

입니다. `GetRandomPatrolPoint`는 `Target` 키를 쓰고, `MoveTo`도 `Target`을 사용합니다. 

네 축약 그래프에서는 `Node7` 아래 태스크가 3개만 보였는데, 그건 축약 과정에서 하나가 생략되었거나 덜 붙은 것으로 봐야 합니다. 원본 구조는 **Idle에 태스크 4개**가 맞습니다. 

---

## 2) Chase = Node2

Node2 아래 `Node4`, `Node3`가 있고, 실제 원본에선 Chase 시퀀스가:

```text id="w0dgra"
SetCurrentState(Chase)
→ MoveTo(Player)
→ Wait
```

입니다.
즉:

* 현재 상태를 Chase로 기록
* `Player` 키를 대상으로 추적 이동
* 잠깐 대기

입니다. `MoveTo`의 BlackboardKey가 실제로 `Player`입니다. 

---

## 3) Battle = Node5

Node5는 Battle 시퀀스인데, 현재 자료상 **내용이 비어 있습니다.**
즉 이름과 데코레이터만 있고, 실제 전투 태스크는 아직 안 넣은 상태로 보입니다. 

그래서 지금은 Battle을 “공격 확장 자리”로 이해하면 됩니다.

---

## 4) Death = Node6

Node6 아래 `Node8` 하나가 있고, 실제 원본에선 Death 시퀀스는:

```text id="2no3au"
Wait
```

하나만 있습니다.
즉 사망 상태에 들어가면 별다른 행동 없이 대기시키는 매우 단순한 상태입니다.

# Decorator 네 개는 뭐냐

네가 붙인 `Node12 ~ Node15`는 전부 **Decorator**입니다.

이 Decorator들은 모두 `CurrentState`를 검사합니다.

* `Idle?`
* `Chase?`
* `Battle?`
* `Death?`

즉 Selector 아래 자식 시퀀스가 실행되기 전에:

> “현재 블랙보드의 `CurrentState` 값이 이 상태와 맞는가?”

를 검사하는 조건문입니다.

# 그래서 BT_Zombie를 어떻게 이해하면 되냐

딱 이 문장으로 이해하면 됩니다.

> **`BT_Zombie`는 `CurrentState`라는 Blackboard 값을 보고 Idle / Chase / Battle / Death 중 하나의 Sequence를 선택해서 실행하는 상태 기반 Behavior Tree다.**

# 네가 직접 정리하면 이런 그림이다

## 최종 해석도

```text id="h6onvf"
Root
→ Selector
   → [Decorator: CurrentState == Idle]
      Idle Sequence
      → SetCurrentState
      → GetRandomPatrolPoint
      → MoveTo(Target)
      → Wait

   → [Decorator: CurrentState == Chase]
      Chase Sequence
      → SetCurrentState(Chase)
      → MoveTo(Player)
      → Wait

   → [Decorator: CurrentState == Battle]
      Battle Sequence
      → (아직 비어 있음 / 공격 넣을 자리)

   → [Decorator: CurrentState == Death]
      Death Sequence
      → Wait
```

이게 현재 구조입니다.

# 여기서 가장 중요한 포인트

## 1. Selector는 “상태 선택기”

`Node1`은 그냥 분기점이 아니라,
**현재 상태값을 보고 어떤 Sequence를 돌릴지 고르는 선택기**입니다. 

## 2. Blackboard의 핵심은 `CurrentState`

이 BT는 사실상 `CurrentState` 중심 설계입니다.
Idle/Chase/Battle/Death 데코레이터가 전부 여기에 묶여 있습니다. 

## 3. 실제 이동용 키는 두 개

* `Player`: 추적 대상
* `Target`: 순찰 목적지

즉 Chase는 `Player`, Idle은 `Target`을 씁니다.

* `Player`

---

# 2. Idle 브랜치가 실제로 도는지 확인

현재 가장 쉬운 건 Idle입니다.

Idle 흐름은:

```text id="7xudk0"
SetCurrentState(Idle)
→ GetRandomPatrolPoint
→ MoveTo(Target)
→ Wait
```

여기서 막히는 곳이 보통 둘 중 하나입니다.

## A. `GetRandomPatrolPoint` 실패

* `FinishExecute(false)`가 날 수 있음
* 그러면 Idle Sequence가 중간에 끊김

## B. `MoveTo(Target)` 실패

* `Target`이 없거나
* NavMesh가 없거나
* 도달 불가 위치면 실패

---

# 바로 넣을 로그

## `GetRandomPatrolPoint`

여기 블루프린트 태스크 안에서 print string 또는 로그로 확인:

* ControlledPawn 위치
* RandomLocation 값
* ReturnValue 성공 여부

즉 확인 포인트:

* 랜덤 위치가 실제로 구해졌는가
* `FinishExecute(true)`가 되는가

---

## `BTTask_SetPlayer`

이것도 확인:

* `GetAllActorsOfClass` 결과가 1개 이상인가
* Player 키에 실제 오브젝트가 들어가는가

---

# 3. NavMesh 확인

이제 더 중요합니다.

### 바로 확인

에디터에서 `P` 눌러서 네비 영역이 초록색으로 보이는지.

없으면:

* `GetRandomReachablePointInRadius` 실패 가능
* `MoveTo` 실패 가능

즉 BT는 돌지만 좀비는 제자리일 수 있습니다.

# 4. BT 디버거로 어디서 멈추는지 보기

이 단계가 제일 실전적입니다.

플레이 중:

* `BT_Zombie` 열기
* 상단 디버그 대상에서 현재 좀비 선택

그럼 초록색으로 현재 실행 중인 노드가 보입니다.

### 여기서 보면 되는 것

* Idle Sequence까지 들어가는가?
* `GetRandomPatrolPoint`에서 멈추는가?
* `MoveTo`에서 실패하는가?
* 아예 아무 브랜치도 안 타는가?

이 한 번 보면 거의 바로 원인 나옵니다.

---

# 네 상황에서 가장 의심되는 순서

## 1순위: `CurrentState` 흐름 문제

네 BT는 상태 기반 분기라 `CurrentState`가 중요합니다.
그런데 구조상 약간 자기모순 가능성이 있습니다.

예를 들면:

* Selector가 `CurrentState == Idle`일 때만 Idle 브랜치 진입
* 그런데 Idle 브랜치 안에서 `SetCurrentState(Idle)`를 함

이러면 **“들어가야 값을 세팅하는데, 들어가기 전에 이미 값이 맞아야 한다”**는 문제가 생길 수 있습니다.

즉 초기값이 Blackboard에 세팅되지 않으면 처음부터 브랜치를 못 탈 수 있습니다.

### 해결 방향

* 시작 시 Blackboard의 `CurrentState`를 `Idle`로 초기화
* 또는 Decorator 없이 기본 브랜치를 하나 열어둠

이건 지금 꽤 강하게 의심됩니다.

---

## 2순위: `Target` 미설정 또는 NavMesh 실패

Idle에서 이동이 안 되는 가장 흔한 이유입니다.

---

## 3순위: `Player` 키 미설정

Chase가 안 되는 원인입니다.

---

# 아주 중요한 구조 체크

지금 BT 설계에서 이 질문이 중요합니다.

## 현재 Blackboard의 `CurrentState` 초기값이 무엇인가?

네 BT는 Decorator가 상태값 비교로만 열리는데,
초기 Blackboard 값이 비어 있으면 Selector에서 아무 브랜치도 못 탈 수 있습니다.

즉 지금 “BT가 시뮬한다”와 “움직인다” 사이에 남은 가장 핵심 질문은:

> **초기 `CurrentState`가 Idle로 세팅되어 있는가?**

입니다.

---

# 바로 해볼 것

## 방법 A. ZombieAIController `OnPossess()`에서 초기 상태 넣기

가장 직관적입니다.

```cpp id="llw7l0"
#include "BehaviorTree/BlackboardComponent.h"

void ASU_ZombieAIController::OnPossess(APawn* InPawn)
{
    Super::OnPossess(InPawn);

    UE_LOG(LogTemp, Warning, TEXT("ZombieAIController OnPossess: %s"), *GetNameSafe(InPawn));

    if (RunBTAsset)
    {
        const bool bRan = RunBehaviorTree(RunBTAsset);
        UE_LOG(LogTemp, Warning, TEXT("RunBehaviorTree result: %s"), bRan ? TEXT("true") : TEXT("false"));

        if (bRan && GetBlackboardComponent())
        {
            GetBlackboardComponent()->SetValueAsEnum(TEXT("CurrentState"), static_cast<uint8>(EZombieState::Idle));
            UE_LOG(LogTemp, Warning, TEXT("Initial CurrentState set to Idle"));
        }
    }
}
```

이렇게 하면 최소한 Idle 데코레이터는 탈 수 있습니다.

---

## 방법 B. 블루프린트에서 Blackboard 초기값 확인

`BB_Zombie`에서 `CurrentState` 기본값이 `Idle`인지 확인.

---

# `GetNameSafe` 오류에 대해서도 짚으면

네가 쓴 이 줄은 맞습니다.

```cpp id="4m9s6z"
UE_LOG(LogTemp, Warning, TEXT("ZombieAIController OnPossess: %s"), *GetNameSafe(InPawn));
```

이건 정상 패턴입니다.

반면 이건 틀렸습니다.

```cpp id="klyl3i"
// GetNameSafe((UObjectBaseUtility*)RunBTAsset)
```

굳이 캐스팅할 필요가 없습니다. 그냥 이렇게 쓰면 됩니다.

```cpp id="1x6svw"
UE_LOG(LogTemp, Warning, TEXT("RunBTAsset: %s"), *GetNameSafe(RunBTAsset));
```

`RunBTAsset`은 `UBehaviorTree*`이고 `UObject` 계열이라 `GetNameSafe`에 바로 넣으면 됩니다.

# 가장 큰 문제 2: Idle의 `GetRandomPatrolPoint`가 `Target`이 아니라 `SelfActor`에 쓰고 있음

  으로 바꾸세요.

# 큰 문제 3: `CurrentState` 데코레이터가 전부 `UNKNOWN!`로 보임

Decorator들을 보면 전부:

* `CurrentState Is Equal To UNKNOWN!`

로 표시됩니다. `Idle?`, `Chase?`, `Battle?`, `Death?` 모두 그렇습니다. 

이건 보통 **Blackboard의 `CurrentState` 키 타입과 비교값 타입이 제대로 안 맞는다**는 신호입니다.

가능성:

* Blackboard `CurrentState`가 `Enum` 타입이 아님
* Enum asset/class 연결이 꼬였음
* 데코레이터는 enum 비교를 기대하는데 현재 키가 int/bool/object 등 다른 타입임

## 확인

`BB_Zombie`에서 `CurrentState` 키를 열어서:

* 타입이 **Enum** 인지
* 그 enum이 **`EZombieState`** 인지

확인하세요.

이게 안 맞으면 Selector가 네 분기 중 아무 데도 못 들어갈 수 있습니다.
