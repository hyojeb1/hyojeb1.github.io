# 🚀 아름다움을 향한 여정: 디테일 깎기

포트폴리오 블로그의 뼈대를 세우고 나니, 이제 개발자의 고질병인 '디테일 집착'이 발동하기 시작했습니다. 화면에 그려지는 픽셀 하나하나의 아름다움을 추구하는 그래픽스 프로그래머로서, 기본 테마의 밋밋함을 그냥 두고 볼 수는 없었죠.

하지만 갓 태어난 Astro 블로그는 호락호락하지 않았습니다. 아바타 이미지가 증발하고, 마크다운 렌더링이 꼬이고, 심지어 터미널에서는 외계어가 쏟아졌습니다. 이번 포스팅은 그 험난했던 트러블슈팅의 기록입니다.

## 🦅 사건 1: 제비 아바타 실종과 이미지 경로의 철학

블로그의 마스코트이자 저의 페르소나인 '헤드셋을 쓴 제비' 픽셀 아트를 야심 차게 올렸습니다. 그런데 웬걸? 엑스박스가 뜨면서 아바타가 렌더링되지 않았습니다.

처음에는 확장자 문제인 줄 알았습니다. 
"아, 테마 기본 캐릭터들은 최신 포맷인 `.webp`인데, 내 제비는 근본 `.png`라서 안 나오는 건가?"

:::swallow
하지만 브라우저가 `.png`를 못 읽을 리가 없잖아요? 진짜 범인은 파일 확장자가 아니라 **'Astro의 이미지 경로 철학'**에 있었습니다.
:::

Astro는 이미지를 다루는 두 가지 명확한 기준이 있었습니다.
1. **`public/` 폴더:** 절대 경로(`/`)를 쓰며, Astro가 전혀 건드리지 않고 원본 그대로 웹에 배포합니다.
2. **`src/` 폴더:** 상대 경로(`../../`)를 쓰며, 빌드 과정에서 Astro가 화질 최적화와 압축을 빡세게 돌립니다.

제비 아바타는 `src/content/`에 얌전히 들어있었는데, 설정 파일에서는 `/jebi_avatar.png`라고 절대 경로로 부르고 있었으니 당연히 찾을 수가 없었던 겁니다. 

> [!info] 해결책: 중복을 혐오하는 자의 선택
> 파일을 `public/`과 `src/` 양쪽에 두는(DRY 위배!) 끔찍한 짓 대신, Astro의 철학을 존중하기로 했습니다. 아바타 이미지는 `public/`으로 몰아주고, 블로그 본문에서 쓰는 스크린샷들은 `src/content/`에 두고 철저하게 **상대 경로**로 접근하여 최적화의 꿀을 빨기로 했습니다.

## 🧱 사건 2: 마크다운 파서(Parser)와의 기싸움

마크다운은 참 편하지만, 가끔 말귀를 못 알아먹을 때가 있습니다. 특히 한국어를 쓸 때 말이죠.

### 1) "이미지를 나란히 두고 싶은데 왜 자꾸 세로로 쌓는 거야!"
마크다운 기본 문법은 이미지를 벽돌처럼 세로로만 쌓습니다. 그래서 HTML의 Flexbox를 직접 주입했습니다.

```html title="가로 정렬 Flexbox (엔터 주의!)"
<div style="display: flex; gap: 10px;">

<div style="width: 50%;">

![이미지1](../../img1.png)

</div>
...

```

### 2) 한국어 조사와 굵기(`**`)의 환장 콜라보

`**픽셀 아트**입니다`라고 썼더니, 조사가 찰싹 붙어버리는 바람에 파서가 단어의 끝(Word Boundary)을 인식하지 못하고 굵기 적용을 씹어버렸습니다.
해결책은? 눈에 보이지 않는 투명 공백 문자 `&#8203;`를 사이에 찔러 넣어서 컴퓨터를 완벽하게 속였습니다. ==딸깍딸깍!==

## 👽 사건 3: 윈도우 인코딩과 터미널 외계어 출몰

```text title="VS Code 터미널"
쒓렇濡?蹂, 留뚮뱾湲

```

태그 라우팅 문제를 고치고 빌드를 돌렸더니 터미널이 크툴루 신화의 주문을 외우기 시작했습니다.
원인은 영원한 숙적, **인코딩 충돌**입니다. Astro 빌드 도구는 UTF-8(유니코드)로 예쁘게 말을 거는데, 윈도우 터미널(PowerShell)은 아직도 90년대 CP949 감성에 젖어있었기 때문이죠.

해결은 간단했습니다. 터미널에 `chcp 65001`을 입력하여 즉시 UTF-8로 정신 개조를 시켜주었습니다.

## 🎮 사건 4: 슬더스 20승천 감성, Neo둥근모 폰트 이식 작전

이 블로그의 화룡점정! D2Coding도 좋지만, 역시 픽셀 아트 제비 아바타에는 레트로한 도트 감성의 **Neo둥근모** 폰트가 제격이었습니다.

하지만 VS Code는 최신 Tailwind v4 문법(`@theme`)을 이해하지 못하고 빨간 에러 줄을 융단폭격하기 시작했습니다. 거기에 `@import` 순서 에러까지!

> [!warning] 폰트 적용이 씹혔다니!
> 오로라 엔진에서 텍스처 셰이더 바인딩이 누락된 것만큼 킹받는 상황이었습니다. 외부 CDN 로드 타이밍이나 변수 매핑이 꼬인 것 같았죠.

결국, 깔끔하게 정면 돌파를 선택했습니다. `global.css` 최상단에 `@font-face`를 직접 때려 박았습니다.

```css title="src/styles/global.css"
/* Neo둥근모: 레트로 감성 & 가독성 동시 확보 */
@font-face {
  font-family: 'NeoDunggeunmo';
  src: url('[https://fastly.jsdelivr.net/gh/projectnoonnu/noonfonts_2001@1.3/NeoDunggeunmo.woff](https://fastly.jsdelivr.net/gh/projectnoonnu/noonfonts_2001@1.3/NeoDunggeunmo.woff)') format('woff');
  font-weight: normal;
  font-style: normal;
  font-display: swap;
}

```

여기서 핵심은 `font-display: swap;` 입니다. 폰트가 다운로드되는 찰나의 순간에도 화면이 하얗게 멈추지 않도록 시스템 폰트를 먼저 띄워주는 최적화 옵션입니다. 그래픽스 프로그래머로서 1프레임의 버벅임도 용납할 수 없으니까요.
